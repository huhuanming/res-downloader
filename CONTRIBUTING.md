# 感谢您对res-downloader的关注与贡献

为了提高沟通和Review效率，请在提交Pull Request前阅读以下规范。

### 1. 提交前须知

对于Bug修复和文档更新，可直接提交PR。

对于新功能、架构调整或大型重构，为避免贡献者投入大量时间后因方向不符而无法合并，建议先创建Issue进行讨论，并获得确认后再开始开发。

未经讨论的大型功能PR可能不会被Review或被要求拆分后重新提交。

### 2. 一个PR只处理一个问题

请确保每个Pull Request仅解决一个独立问题，保持PR小而专注，更有利于快速Review和合并。

✅ 推荐：

* fix: 修复下载失败问题
* fix: 修复Windows路径兼容问题
* feat: 新增代理配置功能

❌ 不推荐：

* 修复下载问题、重构代码
* 新增功能、更新文档、修改样式
* 多个无关问题放在同一个PR

### 3. 标题约定

```text
<类型>: <描述>
```

| 类型       | 说明              |
| -------- | --------------- |
| feat     | 新增功能            |
| perf     | 性能优化            |
| fix      | 修复问题            |
| refactor | 代码重构，不改变现有功能    |
| test     | 测试相关（新增或修改测试用例） |
| docs     | 文档、注释或示例更新      |
| ci       | CI/CD、自动化流程相关修改 |
| chore    | 依赖更新、配置调整等维护工作  |

例如：

```text
feat: 新增代理配置功能
fix: 修复Windows下下载失败问题
perf: 优化大文件下载速度
refactor: 重构下载任务调度逻辑
test: 增加断点续传单元测试
docs: 更新安装说明文档
ci: 优化GitHub Actions构建流程
chore: 升级依赖版本
```

### 4. 提交前

请确保：

* 代码能够正常编译
* 已完成基本测试
* 不包含无关修改
* 文档已同步更新（如有需要）

### 5. 审核流程

由于维护者时间有限，PR Review和Issue回复可能存在延迟。

请耐心等待，不要重复提交相同内容或频繁催促。
